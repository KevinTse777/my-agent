本文档用于沉淀 `docs/上线计划.md` 第五阶段当前已完成的前七小步，重点记录 API 入口限流、登录失败保护、工具超时控制、会话级串行保护、用户级并发控制、日志脱敏与审计日志的设计、实现边界与验收方式。

## 2. 本次为什么先做 API 限流

在第四阶段完成 Kafka、独立 worker、最小重试和 DLQ 后，系统已经具备了异步消费能力，但 API 入口仍然缺少最基本的流量保护。

如果这一层不先补上，当前系统仍然存在明显上线风险：

- 任意来源可以持续冲击 `POST /auth/login`
- 公共调试入口 `POST /chat/agent` 容易被直接打满
- 登录后的高耗时写接口可以被短时间重复提交
- API 线程和后续 Kafka / worker 链路会一起承压

相比“用户级并发控制”“会话级串行控制”这类需要进一步跨 API / worker / 共享存储协同的能力，API 限流更适合作为第五阶段第一步：

- 收益直接
- 改动面集中
- 不破坏现有接口结构
- 可以单独验收和单独回滚

---

## 3. 本次目标

本次目标是给高风险写接口增加最小入口保护，而不是一次性做完整流量治理平台。

完成标准如下：

- 高风险写接口命中限流时返回 `429`
- 不修改现有接口入参和成功响应结构
- 默认支持 Redis 共享计数，未配置 Redis 时本地仍可运行
- 健康检查和只读查询接口暂不误伤
- 测试可重复执行

---

## 4. 实际完成的内容

## 4.1 新增 API 限流服务

新增文件：

- `backend/app/services/rate_limit_service.py`

主要能力：

- 定义高风险接口限流规则
- 解析客户端标识（优先 `X-Forwarded-For`，其次 `request.client.host`）
- 默认优先使用 Redis 计数
- Redis 不可用时回退到 `InMemory`

## 4.2 接入全局中间件

调整文件：

- `backend/app/main.py`

当前行为：

- 请求进入 API 后先注入 `request_id`
- 若命中高风险限流规则，则直接返回 `429 Too Many Requests`
- 响应附带：
  - `Retry-After`
  - `X-RateLimit-Limit`
  - `X-RateLimit-Remaining`
- 请求日志中增加 `limited=true/false`

## 4.3 新增配置项

调整文件：

- `backend/app/core/config.py`
- `.env.example`

新增配置：

- `API_RATE_LIMIT_ENABLED`
- `API_RATE_LIMIT_WINDOW_SECONDS`
- `API_RATE_LIMIT_AUTH_MAX_REQUESTS`
- `API_RATE_LIMIT_CHAT_MAX_REQUESTS`
- `API_RATE_LIMIT_TASK_CREATE_MAX_REQUESTS`

## 4.4 当前纳入保护的接口

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /chat/agent`
- `POST /chat/agent/session`
- `POST /chat/agent/session/stream`
- `POST /chat/tasks`

## 4.5 新增登录失败保护

新增文件：

- `backend/app/services/auth_security_service.py`

当前行为：

- 登录前会先检查当前邮箱是否处于临时锁定期
- 同一邮箱在窗口期内连续失败达到阈值后，会被临时锁定
- 成功登录后会清空对应失败计数与锁定状态
- 被锁定时返回 `429 Too Many Requests`，并附带 `Retry-After`

当前配置项：

- `AUTH_LOGIN_MAX_FAILED_ATTEMPTS`
- `AUTH_LOGIN_ATTEMPT_WINDOW_SECONDS`
- `AUTH_LOGIN_LOCK_SECONDS`

## 4.6 新增工具超时控制

新增文件：

- `backend/app/tools/tool_runtime.py`

当前行为：

- 为 tool 调用新增统一 timeout 包装器
- `web_search_tool` 超时后返回受控超时结果，而不是无限阻塞：
  - `count=0`
  - `sources=[]`
  - `error=timeout`
- `calculator_tool` 也接入同一 timeout 包装器，超时则直接失败

当前配置项：

- `TOOL_CALL_TIMEOUT_SECONDS`
- `CALCULATOR_TOOL_TIMEOUT_SECONDS`
- `WEB_SEARCH_TOOL_TIMEOUT_SECONDS`

## 4.7 新增会话级串行保护

调整文件：

- `backend/app/services/task_store.py`
- `backend/app/services/chat_service.py`

当前行为：

- API 创建聊天任务前，会先检查当前用户当前 `session_id` 是否已有活跃任务
- 活跃任务定义为：`queued` 或 `running`
- 若已存在活跃任务，则返回 `409 Conflict`
- 不同会话仍可继续并行提交

当前边界：

- 这一步先做 API 入口保护，优先减少同会话重复提交流入系统
- 这还不是完整的分布式会话锁；更严格的跨 worker 串行约束仍可留到后续再补

## 4.8 新增用户级并发保护

调整文件：

- `backend/app/services/task_store.py`
- `backend/app/services/chat_service.py`

当前行为：

- API 创建聊天任务前，会先检查当前用户活跃任务总数
- 活跃任务定义为：`queued` 或 `running`
- 当活跃任务数达到 `USER_ACTIVE_TASK_LIMIT` 后，再提新任务会返回 `429 Too Many Requests`
- 该限制只作用于当前用户，不影响其他用户继续提交任务

当前边界：

- 这一步先做用户级总量保护，优先防止单用户同时堆积过多会话任务
- 它与上一小步的会话级串行保护互补，但仍不是更复杂的配额/优先级系统

## 4.9 新增敏感日志脱敏

调整文件：

- `backend/app/core/logging.py`

当前行为：

- 日志系统新增统一脱敏格式器
- 默认会脱敏以下常见敏感值：
  - 邮箱
  - `password`
  - `access_token`
  - `refresh_token`
  - `Authorization` / `Bearer token`
  - JWT 形态 token
- 现有日志调用点无需逐处修改即可自动受益

当前边界：

- 这一步优先处理常见明文敏感信息，先降低直接泄露风险
- 更细粒度的字段级结构化日志和审计事件仍可在后续继续补

## 4.10 新增最小审计日志

新增文件：

- `backend/app/services/audit_store.py`
- `backend/app/services/audit_service.py`

当前行为：

- 新增 `audit_store` 抽象，默认支持 `Postgres` 与 `InMemory`
- 当前优先记录以下关键事件：
  - `auth.register`
  - `auth.login`
  - `auth.refresh`
  - `auth.logout`
  - `chat.task.create`
  - `chat.task.failed`
- 审计事件当前会关联：
  - `user_id`
  - `request_id`
  - 最小业务上下文（如 `task_id`、`session_id`、`refresh_session_id`）

当前边界：

- 这一步先补关键动作留痕，优先满足“出了问题能回看关键动作”
- 暂未扩展到完整的审计查询接口或更全面的事件覆盖

---

## 5. 当前边界

这一步有意只做“入口限流”，暂未一次性覆盖以下能力：

- 用户级并发控制
- 会话级串行控制
- worker 侧并发数治理
- 审计日志
- 敏感日志脱敏

这一步之后仍未继续展开的内容包括：

- 更细粒度的设备 / IP / 风险画像登录策略
- 图形验证码或二次验证
- 登录审计日志
- 更完整的 tool retry / circuit breaker
- 更严格的跨 worker 会话锁
- 更细粒度的用户配额 / 优先级策略
- 更完整的结构化审计日志
- 审计日志查询与管理接口

这些会继续留在第五阶段后续小步推进。

## 6. 测试与验证

本次新增的最小验证包括：

```bash
python -m compileall -q backend/app
cd backend
python -m pytest -q app/tests/test_api.py -k "rate_limit"
```

重点覆盖：

- 连续登录请求会在阈值后返回 `429`
- 连续调用公共聊天入口会在阈值后返回 `429`
- `GET /health` 不受本轮限流影响
- 同一邮箱连续登录失败达到阈值后会被临时锁定
- 登录成功后会清空失败计数
- 锁定期过后可重新登录
- `web_search_tool` 超时后会返回受控超时结果
- `calculator_tool` 超时时会按统一语义报错
- 同一会话存在 `queued/running` 任务时会拒绝再次提交
- 不同会话仍可并行提交任务
- 单用户活跃任务数达到阈值后会拒绝继续提交
- 不同用户互不影响
- 日志中的常见邮箱、password、token 已默认脱敏
- 注册、登录、刷新、登出、任务创建、任务失败已具备最小审计留痕
