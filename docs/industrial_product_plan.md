# StudyMate Agent 工业化落地计划书（v1.0）

## 1. 文档目的

本计划书用于将当前 `my-agent` 项目从“可用 MVP”升级为“可落地、可维护、可扩展”的工业级产品，并明确：
- 业务目标与成功标准
- 技术路线与阶段计划
- 研发与上线的开发边界
- 质量与安全红线
- 验收方式与风险控制

本文件是团队后续开发、评审、排期与验收的统一基线。

---

## 2. 当前基线（As-Is）

### 2.1 已具备能力
- 前后端分离：Vue 3 + Vite，FastAPI 后端
- Agent 主链路：单轮、会话、流式会话
- 工具调用：计算器、Web 搜索
- 会话记忆：`Redis + Postgres` / `Postgres` / `InMemory` 回退
- 基础日志与接口结构已建立

### 2.2 当前主要短板
- `agent_service` 职责集中，维护与测试成本高
- 长任务主要在请求链路内执行，扩展能力有限
- 缺少任务级可观测性（端到端 tracing、成本统计）
- 缺少明确 SLA/SLO 与容量规划
- 缺少工业级发布规范（灰度、回滚、变更审计）

---

## 3. 产品目标（To-Be）

## 3.1 总体目标
在 3-4 个月内建设可用于真实业务试运行的 Agent 平台能力，满足“稳定、可追踪、可治理、可扩展”的生产要求。

### 3.2 成功标准（关键结果）
- 可用性：核心 API 月度可用性 >= 99.9%
- 性能：P95 首 token 时间 <= 2.5s（常规查询）
- 稳定性：核心接口错误率 <= 1%
- 成本：单请求 token 成本可统计，异常增长有告警
- 交付：每周可发布，支持快速回滚（< 10 分钟）
- 安全：密钥零明文入库，关键操作可审计

---

## 4. 目标架构（目标态）

### 4.1 架构分层
- 接入层：`routers`（鉴权、限流、参数校验、request_id 注入）
- 编排层：`agent orchestration`（流程、策略、路由）
- 能力层：`tools`（标准输入输出、超时、重试、熔断）
- 状态层：会话上下文、任务状态、审计日志
- 观测层：日志、指标、链路追踪、告警

### 4.2 执行模式
- 短任务：同步返回（当前链路保留）
- 长任务：异步任务队列 + worker 执行
- 前端：轮询或 WebSocket 获取任务状态与流式结果

### 4.3 部署模式
- 环境分层：`dev` / `staging` / `prod`
- 容器化部署：API 与 worker 分离扩缩容
- 存储分层：Postgres（持久）+ Redis（缓存/队列）

## 5. 工程化里程碑与排期（建议 16 周）

## Phase 0（第 1-2 周）：治理基线与项目规范
目标：先统一规则，再提速开发。

交付：
- 统一目录职责与编码规范（service/tool/repo/config）
- 环境变量分级与密钥管理规范
- CI 基线（lint + unit test + build）
- 变更模板（PR 模板、发布说明模板）

验收：
- 新增代码符合目录职责
- main 分支受保护，CI 失败禁止合并

## Phase 1（第 3-5 周）：单体内重构与质量建设
目标：提升可维护性与可测试性。

交付：
- 拆分 `agent_service`：factory/runner/parser/memory manager
- 统一异常模型与错误码
- Tool 执行包装器（timeout/retry/circuit breaker）
- 单元测试与关键集成测试补齐

验收：
- 核心链路测试覆盖达标（建议 >= 70%）
- 功能回归通过（单轮/会话/流式）

## Phase 2（第 6-9 周）：可观测性与稳定性强化
目标：让问题可发现、可定位、可恢复。

交付：
- 统一结构化日志（含 request_id/session_id/task_id）
- 核心指标：QPS、P95、错误率、tool 成功率、token 成本
- OpenTelemetry tracing（至少贯通 API -> Agent -> Tool）
- 告警规则（错误率、延迟、成本突增）

验收：
- 任一失败请求可在 5 分钟内定位失败节点
- 告警触发与恢复流程可演练

## Phase 3（第 10-13 周）：异步任务与水平扩展
目标：支撑长任务与更高并发。

交付：
- 任务中心：提交、查询、取消、重试
- 队列 + worker 执行模型
- 幂等与去重策略（task key）
- API 与 worker 独立扩容策略

验收：
- 10x 任务峰值下核心链路可用
- Worker 单点故障不影响整体服务可用性

## Phase 4（第 14-16 周）：生产治理与试运行
目标：具备真实上线能力。

交付：
- 灰度发布、回滚预案、版本审计
- 运行手册（故障处理、值班、SOP）
- 安全加固（密钥轮转、最小权限、审计）
- 业务试运行与复盘报告

验收：
- 完成至少 2 次灰度 + 回滚演练
- 试运行期间满足 SLA/SLO 目标

## 6. 开发边界（Scope & Guardrails）

以下边界作为本项目研发红线与取舍依据。

### 6.1 In Scope（当前阶段必须做）
- 围绕学习助手主场景的 Agent 能力建设
- 单模型优先（先稳定，再多模型扩展）
- 核心 Tool 标准化（入参、出参、失败语义）
- 会话记忆可靠性与一致性优化
- 可观测性、测试、CI/CD、发布治理

### 6.2 Out of Scope（当前阶段不做）
- 通用“平台级”多租户计费系统
- 复杂低代码编排器 UI
- 全量自研向量检索平台
- 早期引入过多 Agent 角色（避免过度设计）
- 未验证场景的大规模微服务拆分

### 6.3 技术边界
- 后端主框架保持 FastAPI，不进行框架迁移
- Agent 生态保持 LangChain 路线，不同时并行多套框架
- 存储优先 Postgres + Redis，不新增第三种核心存储
- 所有外部依赖必须有降级与超时策略

### 6.4 质量边界
- 新增核心功能必须携带测试（单测或集成测试）
- 关键接口必须有可观测埋点
- 禁止“无回滚方案”的高风险上线
- P0/P1 缺陷未关闭不得发版

### 6.5 安全边界
- 密钥仅可通过环境变量或密钥管理系统注入
- 禁止在日志输出敏感信息（API key、用户隐私）
- 外部工具调用必须限制超时、域名、重试次数
- 关键管理操作必须可审计

## 7. 非功能性指标（NFR）

### 7.1 性能
- P50 响应时延 <= 1.2s（短问答）
- P95 响应时延 <= 4.0s（非检索复杂问题）
- 流式首 token <= 2.5s

### 7.2 可用性
- 月度可用性 >= 99.9%
- 核心接口错误率 <= 1%

### 7.3 可扩展性
- API 与 worker 支持独立水平扩容
- 队列堆积可监控并可自动告警

### 7.4 可维护性
- 核心模块覆盖率 >= 70%
- 接口与异常语义有文档与示例

## 8. 研发流程与协作机制

### 8.1 分支与发布策略
- 分支模型：`main`（稳定） + `feature/*`（开发）
- 版本语义：`MAJOR.MINOR.PATCH`
- 发布节奏：每周小版本，双周稳定版本

### 8.2 代码评审标准
- 至少 1 名 reviewer
- 必须检查：边界条件、异常处理、可观测性、测试
- 禁止超大 PR（建议 < 500 行有效改动）

### 8.3 测试策略
- 单测：纯逻辑、工具包装器、异常分支
- 集成测试：接口 + memory + tool 调用
- 冒烟测试：发布后关键路径自动验证

---

## 9. 风险清单与应对

### R1：模型供应商波动导致稳定性下降
- 预案：调用超时、重试、熔断、降级回复模板

### R2：外部搜索工具不稳定或成本异常
- 预案：缓存热门查询、限频、失败降级

### R3：长任务阻塞导致整体性能下降
- 预案：引入异步任务中心与 worker 池

### R4：需求扩张过快导致架构失控
- 预案：严格执行 In/Out Scope，按里程碑验收

### R5：观测缺失导致故障定位慢
- 预案：先做日志/指标/trace 再扩功能

- 输出统一错误码和日志字段规范

- 补齐核心链路单测

### Week 3
- Tool 包装器（timeout/retry/circuit breaker）落地

### Week 4
- 完成可观测性 MVP（指标面板 + 基础告警）
- 进行一次全链路故障演练与复盘

---

## 11. 验收与退出标准

项目阶段验收以“可运行证据 + 指标达成 + 文档完整”三要素为准：
- 可运行证据：演示脚本、测试报告、监控截图
- 指标达成：SLA/SLO 与性能指标满足目标
- 文档完整：架构图、运行手册、回滚方案更新完毕

若连续两个迭代未达成阶段目标，应触发范围收缩与优先级重排。

---

## 12. 计划书维护规则

- 本文档由技术负责人维护，双周更新
- 重大变更需记录“变更原因、影响范围、回滚策略”
- 任何新增需求先判断是否落在 In Scope
