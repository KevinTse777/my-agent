## 2. 本次解决了什么问题

在第三阶段结束时，系统虽然已经有了任务接口和后台 worker，但仍存在一个明显限制：

- API 提交任务时直接依赖进程内 worker
- 队列实现和执行器耦合在同一个对象里
- 很难平滑替换成 Kafka
- 很难把 worker 独立成单独进程

本次完成后，系统已经具备了进一步 Kafka 化和进程化的稳定基础：

- 任务投递与任务执行已经分层
- broker 已有统一抽象
- 默认 `InMemory` 模式仍可本地跑通
- Kafka broker 已有可选实现边界
- worker 已可作为独立进程入口启动
- Kafka 模式下不会再静默回退到进程内存储
- 本地 Kafka + 独立 worker 的最小闭环已可验收

## 3. 本次目标

这一步的目标不是一次性完成整套 Kafka 上线能力，而是先把最难返工的基础拆分清楚：

- 把 broker 从 worker 中拆出
- 保持现有任务接口不变
- 保持默认本地开发体验不变
- 给独立 worker 进程留出正式入口
- 为后续 Kafka 联调、重试、DLQ 做准备

完成标准如下：

- API 不再直接依赖进程内队列实现
- worker 成为纯消费执行器
- `inmemory` 模式继续工作
- `kafka` 模式具备基础配置项和代码路径
- Kafka 模式下共享存储前置条件明确
- 本地单 worker 最小闭环可验收
- 测试通过

## 4. 设计决策

## 4.1 为什么先抽 broker，再接真实 Kafka

因为真正容易返工的，不是“调用 Kafka SDK 的那几行代码”，而是系统边界。

如果不先把抽象理顺，后面很容易出现这些问题：

- API 仍然偷偷依赖本地线程
- worker 与 broker 强耦合
- Kafka 接入时要改一大片业务代码
- 本地开发和线上运行的行为越来越分裂

所以这一步先明确：

- `chat_service` 负责创建任务并投递 broker
- `task_worker` 负责消费并执行任务
- broker 决定消息如何传递

## 4.2 为什么保留 `InMemory` 默认模式

因为当前项目仍然需要保持低成本本地验收能力。

保留 `InMemory` 的价值在于：

- 不依赖 Kafka 也能继续开发
- 不影响现有测试
- 能在回退时快速恢复最小可用链路

## 4.3 为什么独立 worker 入口先用脚本

当前阶段先要验证“API 进程”和“worker 进程”可以解耦，而不是马上引入完整部署编排。

先提供脚本入口，可以更快验证：

- 生命周期是否清晰
- 配置是否完整
- broker 切换是否正确

后续再把它接进 Docker Compose 或正式部署脚本。

## 4.4 为什么先补共享存储约束，再做真实 Kafka 联调

因为 API 和 worker 一旦拆成两个进程，`InMemory` 存储就不再是“本地简化实现”，而会直接变成错误配置：

- API 创建的 task 只存在于 API 进程内存
- worker 消费 Kafka 后拿不到同一份 task 记录
- 会话归属和消息回写也会各写各的内存
- 表面现象通常就是任务长期停留在 `queued`

所以在进入真实 Kafka 联调之前，必须先把这类隐性问题转成显式校验，避免继续在错误配置上排查。

## 4.5 为什么需要本地 direct partition fallback

这次真实联调里，API 发布消息到 Kafka 是成功的，但本地单机 Kafka 的 consumer group 协调器返回了 `CoordinatorNotAvailable`。

这意味着：

- Producer 正常，任务能进入 topic
- Consumer group 却不一定能正常工作
- Worker 可能表面在运行，但 task 会长期停在 `queued`

为了先完成“本地最小可验收闭环”，本次给 worker 增加了一个非常克制的 fallback：

- 只有当 consumer group 协调器不可用时才触发
- 仅回退到单 worker 的分区直读模式
- 配合 task 状态幂等检查，避免重放旧消息时重复执行已完成任务

这样既不破坏现有接口，也能让本地 Kafka 模式继续推进。

## 5.5 新增配置与文档

调整文件：

- `backend/app/core/config.py`
- `backend/app/services/auth_store.py`
- `backend/app/services/chat_store.py`
- `backend/app/services/task_store.py`
- `backend/app/services/agent_service.py`
- `.env.example`
- `backend/README.md`

新增配置项：

- `TASK_BROKER_BACKEND`
- `KAFKA_BOOTSTRAP_SERVERS`
- `KAFKA_CHAT_TASK_TOPIC`
- `KAFKA_CHAT_TASK_CONSUMER_GROUP`

本次补充行为：

- API 启动与独立 worker 启动都会先校验 Kafka 运行时配置
- 当 `TASK_BROKER_BACKEND=kafka` 且缺少 `POSTGRES_URL` 时，服务会直接报错
- `task_store`、`chat_store`、`auth_store`、`memory_store` 不再在 Kafka 模式下静默回退到 `InMemory`
- 当 `TASK_BROKER_BACKEND=kafka` 且当前 Python 环境缺少 `kafka-python` 时，API / worker 会直接报错
- worker 在任务执行前会先检查 task 是否仍为 `queued`，避免回放旧 Kafka 消息时重复执行
- 本地 Kafka consumer group 协调器不可用时，worker 会自动回退到 direct partition 模式
- 新增 `python backend/scripts/run_task_worker.py --check` 运行时检查命令，可在启动前直接看到当前预期消费模式
- 新增 `python backend/scripts/run_task_worker.py --diagnose` 诊断命令，可继续检查 broker 连通性、topic metadata 与 `__consumer_offsets` 可见性
- 新增 `docker-compose.kafka-local.yml`，提供固定的本地 `Postgres + Apache Kafka(KRaft)` 联调基线
- 当 `consumer_mode=consumer_group` 时，`--diagnose` 会显式按“包含 internal topics”的口径检查 `__consumer_offsets`，避免被客户端默认过滤误判
- 对较老版本 `kafka-python`，若客户端 API 不支持该检查参数，`--diagnose` 会回退为 `unknown` 并附带说明，而不会再抛出误导性的错误
- 新增 `backend/scripts/smoke_chat_task_flow.py`，用于本地执行 API -> Kafka -> worker -> Postgres 全链路最小验收
- `PostgresMemoryStore` 已补自动建表，避免 fresh Postgres 环境下 worker 首次执行会话链路时报 `chat_messages` 缺表
- 已完成标准 consumer group 模式下的全链路 smoke 验收，task 可稳定从 `queued -> running -> succeeded`
- 已补最小重试与 DLQ：失败任务会按 `TASK_WORKER_MAX_RETRIES` 重试，超过次数后发布到 `chat.request.dlq`
- 新增 `backend/scripts/read_chat_task_dlq.py`，用于本地直接查看 DLQ 中的失败任务消息
- 新增 `backend/scripts/smoke_chat_task_dlq_flow.py`，用于本地直接做 DLQ publish/read 最小验收

## 11. 本地最小联调基线

建议按下面顺序验收：

1. 先确认 `.env` 中至少已配置：
   - `TASK_BROKER_BACKEND=kafka`
   - `KAFKA_BOOTSTRAP_SERVERS`
   - `POSTGRES_URL`
2. 启动前先执行：

```bash
docker compose -f docker-compose.kafka-local.yml up -d
python backend/scripts/run_task_worker.py --check
python backend/scripts/run_task_worker.py --diagnose
python backend/scripts/smoke_chat_task_flow.py --base-url http://127.0.0.1:8000
```

预期输出至少包含：

- `broker_backend: kafka`
- `topic: chat.request`
- `consumer_group: <当前 group>`
- `consumer_mode: consumer_group` 或 `direct_partition_fallback`
- `bootstrap_connected: yes|no`
- `topic_exists: yes|no|unknown`
- `consumer_offsets_topic_visible: yes|no|unknown`

3. 若检查结果是 `consumer_group`：
   - 说明当前环境已具备标准 group 消费前置条件，可直接启动独立 worker
4. 若检查结果是 `direct_partition_fallback`：
   - 说明当前环境仍可做本地最小验收
   - 但这不是第四阶段后续能力的最终上线模式，后面仍需继续收 consumer group

常见故障排查：

- 启动即报 `POSTGRES_URL is required`
  - 说明 Kafka 模式下共享存储未配置，API 与 worker 不能跨进程共享 task/chat/auth/memory 数据
- 启动即报 `kafka-python is required`
  - 说明当前 Python 环境缺少 Kafka 依赖
- `--check` 显示 `direct_partition_fallback`
  - 说明本地 Kafka 协调器不可用或不稳定，当前可先按 fallback 验收最小闭环，再继续排查 Kafka 单机配置
- `bootstrap_connected: no`
  - 说明当前 worker 连到 Kafka bootstrap 节点本身就不稳定，先排查 broker 地址、端口与 Kafka 进程状态
- `topic_exists: no`
  - 说明当前 topic metadata 尚不可见，先确认 topic 是否已创建或 broker 是否允许自动建 topic
- `consumer_offsets_topic_visible: no`
  - 说明 consumer group 依赖的内部 topic 状态异常或尚未 ready，优先继续排查本地 Kafka 单机配置

如果当前自建本地 Kafka 一直表现为：

- `bootstrap_connected: yes`
- `topic_exists: yes`
- `consumer_offsets_topic_visible: no`

建议优先切到仓库内 `docker-compose.kafka-local.yml` 提供的固定单机基线，再重新执行：

```bash
python backend/scripts/run_task_worker.py --check
python backend/scripts/run_task_worker.py --diagnose
python backend/scripts/smoke_chat_task_flow.py --base-url http://127.0.0.1:8000
```

若切换到该基线后仍然无法看到 `__consumer_offsets`，再继续进入下一小步排查镜像、端口占用或本地旧数据卷污染。
