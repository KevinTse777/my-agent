# 上线计划第四阶段总结：Kafka 接入与 worker 进程化第一步

## 1. 文档目的

本文档用于沉淀 `docs/上线计划.md` 第四阶段当前已完成的第 1 步，重点记录 broker 抽象、Kafka 可选接入边界、worker 进程化入口以及本次刻意控制的范围。

本阶段对应上线计划中的核心内容是：

- `5.4 Kafka 接入与 worker 进程化`

---

## 2. 本次解决了什么问题

在第三阶段结束时，系统虽然已经有了任务接口和后台 worker，但仍存在一个明显限制：

- API 提交任务时直接依赖进程内 worker
- 队列实现和执行器耦合在同一个对象里
- 很难平滑替换成 Kafka
- 很难把 worker 独立成单独进程

本次完成后，系统已经具备了进一步 Kafka 化和进程化的稳定基础：

- 任务投递与任务执行已经分层
- broker 已有统一抽象
- 默认 `InMemory` 模式仍可本地跑通
- Kafka broker 已有可选实现边界
- worker 已可作为独立进程入口启动

---

## 3. 本次目标

这一步的目标不是一次性完成整套 Kafka 上线能力，而是先把最难返工的基础拆分清楚：

- 把 broker 从 worker 中拆出
- 保持现有任务接口不变
- 保持默认本地开发体验不变
- 给独立 worker 进程留出正式入口
- 为后续 Kafka 联调、重试、DLQ 做准备

完成标准如下：

- API 不再直接依赖进程内队列实现
- worker 成为纯消费执行器
- `inmemory` 模式继续工作
- `kafka` 模式具备基础配置项和代码路径
- 测试通过

---

## 4. 设计决策

## 4.1 为什么先抽 broker，再接真实 Kafka

因为真正容易返工的，不是“调用 Kafka SDK 的那几行代码”，而是系统边界。

如果不先把抽象理顺，后面很容易出现这些问题：

- API 仍然偷偷依赖本地线程
- worker 与 broker 强耦合
- Kafka 接入时要改一大片业务代码
- 本地开发和线上运行的行为越来越分裂

所以这一步先明确：

- `chat_service` 负责创建任务并投递 broker
- `task_worker` 负责消费并执行任务
- broker 决定消息如何传递

## 4.2 为什么保留 `InMemory` 默认模式

因为当前项目仍然需要保持低成本本地验收能力。

保留 `InMemory` 的价值在于：

- 不依赖 Kafka 也能继续开发
- 不影响现有测试
- 能在回退时快速恢复最小可用链路

## 4.3 为什么独立 worker 入口先用脚本

当前阶段先要验证“API 进程”和“worker 进程”可以解耦，而不是马上引入完整部署编排。

先提供脚本入口，可以更快验证：

- 生命周期是否清晰
- 配置是否完整
- broker 切换是否正确

后续再把它接进 Docker Compose 或正式部署脚本。

## 5. 实际完成的内容

## 5.1 新增 broker 抽象层

新增文件：

- `backend/app/services/task_broker.py`

主要能力：

- 定义 `ChatTaskJob`
- 定义 `TaskBroker` 协议
- 提供 `InMemoryTaskBroker`
- 提供 `KafkaTaskBroker`
- 根据配置自动选择 broker 实现

## 5.2 改造任务投递链路

调整文件：

- `backend/app/services/chat_service.py`

当前行为变为：

- API 创建任务记录
- API 将任务发布到 broker
- 不再直接依赖进程内队列对象

这样后续替换 broker 时，不需要重改业务接口层。

## 5.3 收敛 worker 职责

调整文件：

- `backend/app/services/task_worker.py`

当前 worker 职责明确为：

- 消费 broker 中的任务
- 执行带会话聊天链路
- 更新任务状态
- 写回执行结果或错误

也就是说，worker 不再负责定义消息队列本身。

## 5.4 接入 API 生命周期与独立 worker 入口

调整文件：

- `backend/app/main.py`
- `backend/scripts/run_task_worker.py`

当前行为：

- `TASK_BROKER_BACKEND=inmemory` 时，API 进程仍自动启动内置 worker
- `TASK_BROKER_BACKEND=kafka` 时，API 不再启动内置 worker
- 可通过脚本单独启动 worker 进程

## 5.5 新增配置与文档

调整文件：

- `backend/app/core/config.py`
- `.env.example`
- `backend/README.md`

新增配置项：

- `TASK_BROKER_BACKEND`
- `KAFKA_BOOTSTRAP_SERVERS`
- `KAFKA_CHAT_TASK_TOPIC`
- `KAFKA_CHAT_TASK_CONSUMER_GROUP`

## 6. 当前边界与限制

本次已经完成的是“结构准备”，但还没有一次性做完以下内容：

- 真实 Kafka 环境联调
- 重试机制
- DLQ
- 幂等保护
- 任务超时与取消
- Docker Compose 中拆分 API / worker 服务

这些内容不是遗漏，而是刻意留到下一小步，避免当前提交范围过大。
