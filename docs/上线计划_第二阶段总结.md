# 上线计划第二阶段总结：用户与登录体系最小闭环

## 1. 文档目的

本文档用于沉淀 `docs/上线计划.md` 第二阶段已完成的工作，重点记录最小用户体系、鉴权方式、会话绑定关系和验证结果，方便后续继续推进。

本阶段对应上线计划中的核心内容是：

- `5.1 用户与登录体系`

---

## 2. 本阶段解决了什么问题

在本阶段之前，系统虽然已经有了业务层会话和消息，但仍然是匿名模式：

- 没有用户注册与登录
- 没有登录态校验
- 没有会话归属
- 任何人都无法基于用户维度管理自己的会话

本阶段完成后，系统已经具备最小用户闭环：

- 可以注册
- 可以登录
- 可以刷新 token
- 可以退出登录
- 可以获取当前登录用户信息
- 业务会话已绑定 `user_id`
- 业务会话接口和带会话聊天接口已具备权限隔离

---

## 3. 本阶段目标

本阶段目标是把系统从“匿名业务原型”推进到“有用户边界的 Web 服务”。

完成标准如下：

- 补齐最小用户表与登录态表
- 提供最小鉴权接口
- 为业务接口加登录校验
- 为业务会话加用户归属
- 不破坏当前单轮调试接口
- 测试通过

---

## 4. 设计决策

## 4.1 为什么这一阶段先做最小闭环

因为后续 Kafka、Worker、限流、审计都建立在“谁发起了这次请求”之上。

如果没有用户边界，后续实现会有明显缺口：

- 无法绑定 `user_id`
- 无法做权限控制
- 无法形成真实用户使用模型
- 无法解释上线后的安全边界

## 4.2 为什么保留 `/chat/agent`

本阶段只对业务接口和带会话聊天接口加鉴权。

保留：

- `POST /chat/agent`

作为开发调试入口，目的有两个：

- 不影响当前本地排查 Agent 主链路
- 避免前后端在第二阶段被一次性全部锁死

## 4.3 为什么 access / refresh 都走 Bearer + JSON 返回

这是当前阶段最简单、最清晰的方案：

- 前端容易接入
- 接口行为容易验证
- 适合当前项目的最小闭环

后续如果进入更正式上线阶段，再考虑：

- HttpOnly Cookie
- 更严格的 session 管理
- 多端登录控制

## 5. 实际完成的内容

## 5.1 新增用户存储层

新增文件：

- `backend/app/services/auth_store.py`

主要能力：

- 定义 `AuthStore` 抽象
- 提供 `InMemoryAuthStore`
- 提供 `PostgresAuthStore`
- 自动建表并管理用户与 refresh session

本阶段新增的核心业务表概念：

- `app_users`
- `app_user_sessions`

## 5.2 新增鉴权服务

新增文件：

- `backend/app/services/auth_service.py`

主要能力：

- 密码哈希存储与校验
- Access Token 签发
- Refresh Token 签发与刷新
- 登出时撤销 refresh session
- 从 access token 解析当前用户

## 5.3 新增鉴权依赖与接口

新增文件：

- `backend/app/dependencies/auth.py`
- `backend/app/routers/auth.py`

新增接口：

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`
- `GET /me`

## 5.4 将业务会话绑定用户

重构文件：

- `backend/app/services/chat_store.py`
- `backend/app/services/chat_service.py`
- `backend/app/routers/chat.py`

本次调整后：

- 会话包含 `user_id`
- 查询会话列表按当前用户过滤
- 查询消息时校验是否属于当前用户
- 删除会话时校验是否属于当前用户
- 带会话聊天接口会自动将消息写入当前用户的会话

- `AUTH_REFRESH_TOKEN_TTL_SECONDS`

同步更新：

---

## 7. 权限模型

当前采用的是最小权限模型：

- 一个用户只能访问自己的会话
- 一个用户不能查看其他用户的消息
- 一个用户不能删除其他用户的会话
- 使用他人 `session_id` 访问时会被拒绝

这一步完成后，系统的数据边界已经从“匿名会话”升级为“用户拥有会话”。

---

## 11. 本阶段的价值

本阶段的核心价值，是让系统真正拥有了“用户边界”。

现在项目已经从：

- “有业务会话，但没有真实用户”

推进到：

- “用户拥有自己的会话，且业务接口具备权限隔离”

这会显著降低后续实现以下能力的复杂度：

- 个人历史会话页
- 用户维度的异步任务
- 登录态审计
- 限流与安全治理
- 生产部署时的访问控制

---

## 12. 下一阶段建议

建议下一阶段优先进入：

- `5.3 聊天任务异步化`

推荐原因：

- 用户身份和会话归属已经明确
- 现在已经可以把 `user_id`、`session_id`、`request_id` 一起放入任务模型
- Kafka / Worker 的任务结果也能明确写回到正确用户会话

建议最小范围可以是：

- `POST /chat/tasks`
- `GET /chat/tasks/{task_id}`
- `GET /chat/tasks/{task_id}/result`
- 先做“提交任务 + 轮询结果”的最小闭环
