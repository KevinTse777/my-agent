# 上线计划第一阶段总结：业务层会话与消息系统骨架

## 1. 文档目的

本文档用于沉淀 `docs/上线计划.md` 第一阶段已完成的工作，方便后续回顾、继续开发、准备演示或面试表达。

本阶段对应上线计划中的核心内容是：

- `5.2 会话与消息系统`

本次实现遵循的原则是：

- 先建立业务层数据边界
- 先保证可演进，再补用户与异步
- 保持老接口可用，避免一次性重构过大

---

## 2. 本阶段解决了什么问题

在本阶段之前，项目中的“会话”主要是 Agent 记忆层概念：

- `memory_store` 用于给 Agent 提供上下文
- `session_id` 主要服务于多轮对话能力
- 前端会话列表和历史消息主要保存在本地

这会带来几个问题：

- 无法把聊天历史作为正式业务数据管理
- 后续无法自然绑定 `user_id`
- 无法为异步任务、审计日志提供稳定归属
- 容易把“Agent 记忆”和“产品会话历史”混为一谈

本阶段完成后，系统已经具备了最小业务层会话骨架：

- 可以创建业务会话
- 可以查询会话列表
- 可以查询某个会话的消息历史
- 可以删除会话
- 在调用现有会话聊天接口时，消息会同步写入业务层存储

---

## 3. 本阶段目标

本阶段的目标不是直接上线，而是先把后续能力依赖的数据底座立住。

完成标准如下：

- 业务会话与 Agent 记忆明确分层
- 后端提供最小业务会话 CRUD 接口
- 聊天消息可在业务层持久化
- 当前接口和前端链路不被破坏
- 本地可验证、测试可通过

---

## 4. 设计决策

## 4.1 为什么先做业务会话层

因为后续几个关键能力都依赖它：

- 用户登录后，需要把会话绑定到用户
- 历史记录页面，需要查询稳定的业务消息
- 异步任务完成后，需要把结果写回具体会话
- 审计和问题排查，需要明确消息归属

如果不先拆出业务会话层，后面做登录或 Kafka 都容易返工。

## 4.2 为什么不直接复用 `memory_store`

因为它们的职责不同：

- `memory_store`：解决 Agent 推理时如何取上下文
- `chat_store`：解决产品层如何保存会话和消息

两者虽然都和消息有关，但不应该混用。

## 4.3 为什么采用 `Postgres -> InMemory` 回退

这是为了兼顾开发便利和未来上线：

- 配了 `POSTGRES_URL` 时，可以直接使用真实持久化
- 没配时，也能在本地直接开发和验收
- 不会因为数据库暂未准备好而阻塞当前迭代

## 5. 实际完成的内容

## 5.1 新增业务会话存储层

新增文件：

- `backend/app/services/chat_store.py`

主要能力：

- 定义 `ChatStore` 抽象接口
- 提供 `InMemoryChatStore`
- 提供 `PostgresChatStore`
- 根据配置自动选择存储实现

业务层数据结构包括：

- `ChatSession`
- `ChatMessage`

支持的核心操作包括：

- 创建会话
- 确保会话存在
- 查询会话列表
- 查询会话消息
- 追加消息
- 删除会话

## 5.2 新增业务会话接口

在 `backend/app/routers/chat.py` 中新增接口：

- `GET /chat/sessions`
- `POST /chat/sessions`
- `GET /chat/sessions/{session_id}/messages`
- `DELETE /chat/sessions/{session_id}`

这意味着当前后端已经具备最基础的业务会话管理能力。

## 5.3 让现有聊天接口写入业务层消息

在 `backend/app/services/chat_service.py` 中完成接入：

- `POST /chat/agent/session`
- `POST /chat/agent/session/stream`

现在这两个接口除了继续走原有 Agent 记忆链路，还会把：

- 用户消息
- 助手消息
- `tools_used`
- `sources`

同步写入业务层 `chat_store`。

## 5.4 新增配置项

新增环境变量：

- `BUSINESS_CHAT_HISTORY_LIMIT`

用途：

- 控制查询业务消息历史时的最大返回条数

已同步更新：

- `.env.example`
- `backend/README.md`

## 6. 数据与接口边界

## 6.1 业务层与记忆层的边界

记忆层：

- 由 `memory_store` 管理
- 用于给 Agent 提供多轮上下文
- 不直接等价于用户可见的业务历史

业务层：

- 由 `chat_store` 管理
- 用于产品层的会话和历史消息
- 是后续绑定用户、任务、审计的基础

## 6.2 当前会话模型的定位

当前仍然是“匿名单用户开发模式”：

- 会话还没有绑定 `user_id`
- 还没有登录鉴权
- 还没有权限校验

这是刻意控制范围后的结果，目的就是先把底座搭好。

## 10. 当前仍未解决的问题

本阶段有意没有处理以下内容：

- 用户注册、登录、鉴权
- `user_id` 与会话绑定
- 会话权限校验
- 聊天任务异步化
- Kafka / Worker 接入
- 分页、归档、摘要生成
- 审计日志与限流

这些不是遗漏，而是按“先打地基，再扩能力”的顺序留到后续阶段。

## 11. 本阶段的价值

这一步最大的价值，不是多了几个接口，而是把系统的数据边界理顺了。

现在项目已经从：

- “前端本地维护会话 + 后端只负责 Agent 对话”

向：

- “后端开始具备正式业务会话能力”

迈出了一步。

这会直接降低后续实现以下功能的复杂度：

- 登录后我的会话列表
- 历史消息页
- 异步任务结果回写
- 用户维度的数据归属
- 审计和排障

---

## 12. 下一阶段建议

建议下一阶段优先进入：

- `5.1 用户与登录体系`

推荐原因：

- 业务会话层已经存在，可以自然绑定用户
- 登录鉴权是上线前最关键的基础能力之一
- 做完鉴权之后，再推进 Kafka 异步任务链路会更顺

建议最小范围可以是：

- 用户注册
- 用户登录
- JWT 鉴权
- `GET /me`
- 会话绑定 `user_id`
