# 上线计划第三阶段总结：聊天任务异步化最小闭环

## 1. 文档目的

本文档用于沉淀 `docs/上线计划.md` 第三阶段已完成的工作，重点记录聊天任务异步化、任务状态机、worker 执行链路和当前实现边界。

本阶段对应上线计划中的核心内容是：

- `5.3 聊天任务异步化`

---

## 2. 本阶段解决了什么问题

在本阶段之前，系统虽然已经有了用户体系和业务会话，但聊天主链路仍然主要是同步请求：

- API 请求线程直接执行高耗时 Agent 调用
- 长请求会直接占用 API 处理能力
- 没有正式的任务状态管理
- 不能用“提交任务 + 查询结果”的方式解耦前后端

本阶段完成后，系统已经具备最小异步聊天任务闭环：

- 可以创建聊天任务
- 可以查询任务状态
- 可以查询任务结果
- Worker 可以异步消费任务并执行聊天
- 执行结果会同时写回任务结果和业务会话

---

## 3. 本阶段目标

本阶段目标不是一步到位把 Kafka 全接完，而是先把真正难返工的部分稳定下来：

- 任务模型
- 状态机
- API 与 worker 分层
- 结果回写方式
- 用户、会话、任务之间的绑定关系

完成标准如下：

- 有任务存储
- 有任务接口
- 有后台 worker
- 有任务状态流转
- 有任务结果回写
- 测试通过

---

## 4. 设计决策

## 4.1 为什么先做 `InMemory worker`

因为这一阶段最关键的不是“消息发送到 Kafka 的那一行代码”，而是任务模型本身。

如果一开始就强依赖 Kafka，会被基础设施准备、联调和环境问题拖住；而当前项目更需要先把这些核心问题固化：

- task 如何建模
- task 如何绑定 `user_id / session_id / request_id`
- worker 如何执行会话聊天
- 成功和失败如何回写

所以当前阶段采用：

- `InMemory` 队列
- 进程内 worker 线程

先跑通整条业务链路。

## 4.2 为什么结果同时写入任务和业务会话

因为它们职责不同：

- 任务结果：用于查询任务状态与任务输出
- 业务会话：用于历史消息和产品层对话记录

这样后续无论前端走“任务轮询”还是“历史会话”，都能拿到一致数据。

## 4.3 为什么保留同步聊天接口

本阶段新增了异步任务接口，但没有删除原有同步接口。

这是为了：

- 便于开发调试
- 便于对比同步 / 异步行为
- 避免前端和接口一次性全量迁移

## 5. 实际完成的内容

## 5.1 新增任务存储层

新增文件：

- `backend/app/services/task_store.py`

主要能力：

- 定义 `TaskStore` 抽象
- 提供 `InMemoryTaskStore`
- 提供 `PostgresTaskStore`
- 支持任务创建、状态查询、运行中更新、成功更新、失败更新

当前任务模型包含：

- `id`
- `user_id`
- `session_id`
- `input_text`
- `status`
- `request_id`
- `result`
- `error_message`
- `created_at / updated_at / started_at / finished_at`

## 5.2 新增后台 worker

新增文件：

- `backend/app/services/task_worker.py`

主要能力：

- 定义 `ChatTaskJob`
- 提供 `InMemoryTaskWorker`
- 应用启动时自动启动 worker
- worker 从队列消费任务后执行会话聊天
- 根据结果更新任务状态

## 5.3 新增异步任务接口

在 `backend/app/routers/chat.py` 中新增接口：

- `POST /chat/tasks`
- `GET /chat/tasks/{task_id}`
- `GET /chat/tasks/{task_id}/result`

行为说明：

- `POST /chat/tasks`：创建任务并立即返回 `queued` 状态
- `GET /chat/tasks/{task_id}`：查询当前状态
- `GET /chat/tasks/{task_id}/result`：仅在 `succeeded` 时返回结果

## 5.4 接入任务执行链路

在 `backend/app/services/chat_service.py` 中新增：

- 创建异步聊天任务
- 查询任务状态
- 查询任务结果

worker 执行的仍然是带会话聊天链路，因此不会绕开已有的：

- 用户会话校验
- 会话消息写入
- Agent 记忆写入

## 5.5 应用生命周期接入 worker

在 `backend/app/main.py` 中完成：

- 应用启动时启动 worker
- 应用关闭时停止 worker

这样本地运行后，任务接口已经具备完整的后台消费能力。

---

## 6. 当前状态机

当前已落地的任务状态为：

- `queued`
- `running`
- `succeeded`
- `failed`

这是当前阶段的最小可用状态机。

后续如接入更完整的 broker 和 worker 管理，可继续扩展：

- `timeout`
- `cancelled`
- `retrying`

---

## 7. 当前接口边界

本阶段新增的任务接口均需要登录：

- `POST /chat/tasks`
- `GET /chat/tasks/{task_id}`
- `GET /chat/tasks/{task_id}/result`

权限模型：

- 任务绑定到 `user_id`
- 用户只能查询自己的任务
- 任务执行时会继续沿用当前会话权限边界

## 10. 当前仍未解决的问题

本阶段有意没有一次性处理以下内容：

- 真正的 Kafka broker
- 独立部署的 worker 服务
- 重试机制
- 死信队列
- 任务超时控制
- 任务取消
- 流式异步事件推送

这些能力会在后续 Kafka 与 Worker 正式化阶段继续推进。

---

## 11. 本阶段的价值

第三阶段最大的价值，不是简单多了三个接口，而是系统第一次具备了“API 与长任务解耦”的正式模型。

现在项目已经从：

- “用户拥有会话，但聊天主要仍是同步执行”

推进到：

- “API 可以提交任务，worker 异步执行并回写结果”

这会直接降低后续实现以下能力的复杂度：

- Kafka 接入
- Worker 独立部署
- 任务监控与审计
- 失败重试与 DLQ
- SSE / WebSocket 结果推送

---

## 12. 下一阶段建议

建议下一阶段优先进入：

- `6. Kafka 接入方案`

推荐原因：

- 任务模型和状态机已经落地
- API 与 worker 的职责已经明确
- 现在只需要把 broker 从 `InMemory` 队列替换为 Kafka，并把 worker 逐步独立出来

建议最小范围可以是：

- 抽象 broker 接口
- 新增 Kafka producer / consumer 实现
- worker 进程化
- `chat.request` topic 接入

## 13. 一句话总结

第三阶段已经完成“聊天任务异步化最小闭环”，系统从同步聊天服务演进为具备任务模型、后台 worker 和结果轮询能力的在线产品骨架。
