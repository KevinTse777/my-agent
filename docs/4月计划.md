# 4月计划

## 1. 计划目的

本计划面向“求职面试准备”场景，目标不是继续无节制扩功能，而是把当前 `StudyMate Agent` 项目打磨成：

- 能稳定演示
- 能体现工程能力
- 能讲清设计取舍
- 能写进简历并经得起追问

本月工作重点聚焦在“项目质量、可讲述性、可验证性、展示效果”四个方向。

---

## 2. 当前项目基线

项目当前已经具备以下基础能力：

- 前后端分离：`Vue 3 + Vite` / `FastAPI`
- Agent 主链路：单轮对话、会话对话、流式对话
- 工具调用：计算器、Web 搜索
- 会话记忆：`Postgres + Redis -> Postgres -> InMemory` 回退
- 基础文档：项目说明、架构说明、工程边界说明

当前最主要的问题不是“没有功能”，而是：

- 项目亮点还不够集中，不够像一个“面试项目”
- 关键链路的测试、日志、异常处理还不够突出
- 现有文档更偏开发说明，缺少面试展示材料
- 项目故事还不够完整，难以高质量回答“为什么这样设计”

---

## 3. 4 月目标

### 3.1 总目标

在 4 月内把本项目打磨成一个适合用于后端 / AI 应用方向面试展示的代表性项目。

### 3.2 结果标准

到 4 月底，希望至少达到以下标准：

- 可以在 5 分钟内完成一次完整 Demo
- 可以清楚说明系统架构、调用链路和设计边界
- 可以回答关键技术取舍问题
- 关键链路有测试和异常处理支撑
- 简历中可以把该项目提炼成 2-3 条有分量的经历描述

---

## 4. 本月优化方向

## 方向一：补强“项目故事”和面试表达

目标：让项目不只是“能跑”，而是“能讲”。

要做的事情：

- 梳理一版统一项目介绍话术
- 明确项目定位：为什么做学习助手，为什么采用前后端分离
- 明确“瘦身版”的设计原因：为什么删减功能，为什么保留当前主链路
- 总结技术选型依据：
  - 为什么用 FastAPI
  - 为什么用 LangChain
  - 为什么保留工具调用和会话记忆
  - 为什么采用分层结构
- 整理常见追问的回答素材：
  - 如何做异常处理
  - 如何做会话记忆回退
  - 如果搜索工具挂了怎么办
  - 如果请求变慢如何排查

交付物：

- 面试版项目介绍文档
- 面试常见追问问答提纲

---

## 方向二：补强后端工程质量

目标：让项目体现出可靠性和工程意识。

要做的事情：

- 检查 `routers -> services -> tools` 分层是否清晰
- 梳理 `agent_service` 当前职责是否过重
- 明确异常处理策略和返回语义
- 为关键链路增加统一日志字段：
  - `request_id`
  - `session_id`
  - `tool_name`
  - `duration_ms`
- 检查工具调用是否具备基本的超时和失败降级能力
- 检查会话存储回退链路是否可解释、可验证

优先关注文件：

- `backend/app/services/agent_service.py`
- `backend/app/services/chat_service.py`
- `backend/app/services/memory_store.py`
- `backend/app/tools/search_web.py`
- `backend/app/core/logging.py`

交付物：

- 一版更清晰的后端主链路
- 一组更适合在面试中展示的工程亮点

---

## 方向三：补齐关键测试

目标：证明项目不是“演示代码”，而是经过验证的工程实现。

要做的事情：

- 补充聊天接口成功路径测试
- 补充异常路径测试
- 补充会话连续性测试
- 补充 memory store 回退逻辑测试
- 补充流式接口基础格式测试
- 检查工具调用失败时接口是否仍能合理响应

建议优先覆盖：

- `POST /chat/agent`
- `POST /chat/agent/session`
- `POST /chat/agent/session/stream`
- memory fallback
- tool failure fallback

交付物：

- 关键接口测试用例
- 面试可展示的测试思路与质量保障说明

---

## 方向四：强化可观测性和排障能力

目标：能回答“出了问题你怎么查”。

要做的事情：

- 统一请求日志格式
- 补充关键耗时日志
- 明确报错时需要打印哪些上下文
- 为工具调用和会话存储增加可追踪信息
- 输出一份简洁的排障说明：
  - 请求失败看哪里
  - 会话异常看哪里
  - 搜索失败看哪里
  - 流式响应异常看哪里

交付物：

- 更清晰的日志字段规范
- 一份简单排障手册

---

## 方向五：补强展示材料

目标：让项目更适合简历、面试和现场演示。

要做的事情：

- 优化 README，让项目价值和亮点更突出
- 补一版更清晰的架构图
- 补一版调用时序图
- 准备 Demo 脚本：
  - 单轮提问
  - 多轮会话
  - 搜索增强回答
  - 流式响应
- 整理 2-3 个高质量演示问题

交付物：

- 面试版 README 或补充文档
- Demo 脚本
- 架构图与时序图

---

## 方向六：提炼简历价值

目标：把项目成果转化成可写进简历的表达。

要做的事情：

- 提炼项目中的技术关键词：
  - FastAPI
  - LangChain Agent
  - Tool Calling
  - Session Memory
  - Stream Response
  - Vue 3
- 提炼你真正做过的工程动作：
  - 分层设计
  - 接口联调
  - 会话记忆回退
  - 工具调用集成
  - 测试与日志补齐
- 整理 2-3 条“结果导向”的简历描述

交付物：

- 一版简历项目描述
- 一版面试自我介绍中的项目摘要

---

## 5. 4 月执行排期

## 第 1 周：项目梳理与文档补强

重点：

- 梳理项目故事
- 整理面试表达素材
- 补充项目说明文档

计划任务：

- 完成面试版项目介绍
- 整理系统架构与调用链路说明
- 输出一版常见追问提纲

验收标准：

- 能在 3-5 分钟内完整介绍项目
- 能清楚说明系统结构与模块职责

## 第 2 周：后端质量补强

重点：

- 强化主链路代码质量
- 补异常处理与日志

计划任务：

- 梳理 `agent_service`、`memory_store`、`search_web`
- 增加关键日志字段
- 明确异常处理与降级逻辑

验收标准：

- 核心调用链可追踪
- 关键失败场景有可解释行为

## 第 3 周：测试与验证

重点：

- 给关键路径补测试
- 让项目更像“可交付产品”

计划任务：

- 增加接口测试
- 增加会话与回退逻辑测试
- 检查流式接口表现

验收标准：

- 关键接口有测试覆盖
- 可展示测试思路与质量保障方式

## 第 4 周：展示与简历提炼

重点：

- 准备 Demo
- 准备简历和面试表达

计划任务：

- 优化 README / 演示材料
- 输出 Demo 脚本
- 输出简历项目描述
- 模拟面试问答

验收标准：

- 可以稳定演示项目
- 可以把项目浓缩成高质量简历表述

## 6. 本月优先级

### P0：必须完成

- 面试版项目介绍文档
- 核心链路日志与异常处理补强
- 关键接口测试补齐
- Demo 脚本准备
- 简历项目描述提炼

### P1：重要但可顺延

- 更完整的架构图与时序图
- 更细致的排障文档
- 更系统的追问问答题库

### P2：可后续继续优化

- 更深的性能优化
- 更完整的监控指标面板
- 更复杂的任务中心或异步能力

---

## 7. 风险与注意事项

- 不要为了“看起来厉害”而继续堆太多功能
- 不要在 4 月引入过大范围重构，避免影响演示稳定性
- 不要只写文档不补验证，否则面试时容易被追问击穿
- 不要只补代码不整理表达，否则项目价值难以体现

本月原则：

- 先稳定
- 再补强
- 最后包装

---

## 8. 月底验收清单

到 4 月底，至少确认以下事项已完成：

- 项目可以本地稳定运行
- 单轮 / 多轮 / 流式 / 搜索能力都可演示
- 核心接口有基本测试
- 核心链路有日志和异常处理说明
- 有一份适合面试讲解的项目说明
- 有一份简历可直接使用的项目描述

---

## 9. 建议的最终成果

如果本月执行顺利，最终希望形成以下成果包：

- 一个可运行的 Agent 学习助手项目
- 一套面试版技术说明材料
- 一份清晰的 Demo 讲解脚本
- 一组常见技术追问回答提纲
- 一版可以直接写进简历的项目描述

这份计划的核心目标不是“把项目做得更大”，而是“把项目打磨得更适合面试胜出”。
