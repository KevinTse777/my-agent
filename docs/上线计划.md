# StudyMate Agent 上线计划

## 1. 文档目的

本计划用于将当前 `StudyMate Agent` 项目从“本地可运行的 Agent 应用”推进为“可对外提供服务的小型在线产品”。

本计划重点解决以下问题：

- 系统如何具备真实用户使用能力
- 系统如何应对一定规模的并发请求
- 系统如何具备登录、鉴权、历史记录等常规 Web 服务能力
- 系统如何通过 Kafka 建立异步任务处理链路
- 系统如何满足上线所需的安全、监控、部署与运维要求

本计划默认遵循“先可上线，再可扩展”的原则，避免过早复杂化。

---

## 2. 当前项目现状

当前项目已经具备以下基础：

- 前端：`Vue 3 + Vite`
- 后端：`FastAPI`
- Agent 主链路：
  - 单轮对话
  - 会话对话
  - 流式对话
- 工具调用：
  - 计算器
  - Web 搜索
- 会话记忆：
  - `Postgres + Redis`
  - `Postgres`
  - `InMemory` 回退

当前不适合直接上线的主要原因：

- 没有用户注册、登录、鉴权体系
- 没有用户与会话、消息之间的业务绑定关系
- 高耗时 Agent 请求仍主要运行在同步请求链路中
- 缺少任务状态管理，不适合做高并发削峰
- 缺少限流、审计、安全治理和发布体系
- 缺少明确的部署分层和可观测性方案

---

## 3. 上线目标

## 3.1 总体目标

在保留当前 Agent 聊天能力的基础上，将系统升级为一个具备以下特征的 Web 产品：

- 支持真实用户注册、登录和鉴权
- 支持用户拥有自己的聊天会话和历史记录
- 支持异步任务执行，降低高耗时请求对 API 的阻塞
- 支持一定规模并发请求下的稳定运行
- 支持基础安全治理、日志追踪和故障排查
- 支持容器化部署与环境分层

## 3.2 本阶段目标边界

本次上线计划聚焦“小型可上线版本”，不追求一步做到平台化。

本阶段 In Scope：

- 用户系统
- 会话与消息持久化
- Kafka 异步任务链路
- Worker 执行模型
- 基础限流与安全能力
- Docker 化部署
- 日志、监控、告警基础能力

本阶段 Out of Scope：

- 多租户计费系统
- 复杂权限中心
- 大规模微服务拆分
- 全量实时 token 级事件总线
- 复杂的运营后台

## 4. 目标架构

建议采用如下结构：

### 4.1 组件划分

- `frontend`
  - 登录、聊天、历史记录、个人中心
- `api service`
  - 鉴权、会话管理、消息创建、任务创建、结果查询
- `worker service`
  - 消费 Kafka 任务，执行 Agent、工具调用、记忆读写、结果落库
- `postgres`
  - 用户、会话、消息、任务、审计日志等持久化数据
- `redis`
  - 限流、缓存、短期状态
- `kafka`
  - 异步任务与事件流转

### 4.2 请求模式

- 轻量同步接口：
  - 注册、登录、查询会话、查询历史消息、查询任务状态
- 异步处理接口：
  - 聊天请求
  - 高耗时工具调用场景
  - 后续审计与行为事件

### 4.3 核心链路

`前端 -> API -> 鉴权/限流 -> 创建 task -> 写入 Kafka -> Worker 消费 -> 执行 Agent -> 写回 Postgres -> 前端查询或订阅结果`

## 5. 业务能力规划

## 5.1 用户与登录体系

当前状态：已完成第一版最小闭环

已落地：

- 用户注册
- 用户登录
- Access Token 鉴权
- Refresh Token 刷新
- 用户退出登录
- 密码哈希存储
- `GET /me` 登录态校验
- 业务会话绑定 `user_id`
- 业务接口按当前用户做权限校验

需要补齐以下能力：

- 用户注册
- 用户登录
- JWT 鉴权
- 刷新 token
- 用户退出登录
- 密码加密存储
- 登录状态校验

建议接口：

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`
- `GET /me`

设计原则：

- 所有聊天会话必须绑定用户
- 匿名 `session_id` 仅允许开发调试，不作为线上主模式
- 所有业务接口应基于登录用户做权限校验

---

## 5.2 会话与消息系统

当前状态：已完成第一版业务层骨架

已落地：

- 用户会话列表
- 聊天消息持久化
- 首条消息自动生成会话标题
- 历史消息查询
- 会话删除
- 业务会话与 Agent 记忆明确分层

要从“Agent 内部记忆”扩展为“业务层会话系统”。

需要补齐：

- 用户会话列表
- 聊天消息持久化
- 会话标题或摘要生成
- 历史消息分页查询
- 会话删除或归档

建议接口：

- `GET /chat/sessions`
- `POST /chat/sessions`
- `GET /chat/sessions/{id}/messages`
- `DELETE /chat/sessions/{id}`

说明：

- `memory_store` 解决的是 Agent 记忆问题
- `chat_sessions` 和 `chat_messages` 解决的是业务数据持久化问题
- 二者要明确分层，不要混用

---

## 5.3 聊天任务异步化

当前状态：已完成第一版最小闭环

已落地：

- `POST /chat/tasks`
- `GET /chat/tasks/{task_id}`
- `GET /chat/tasks/{task_id}/result`
- 任务状态机：`queued -> running -> succeeded/failed`
- API 创建任务并投递到 worker
- worker 异步执行带会话聊天链路
- 任务结果写回业务会话与任务结果存储
- 当前 broker 采用进程内 `InMemory` 队列
- Kafka 接口边界已预留，后续可替换 broker 实现

当前系统要具备并发承载能力，必须把高耗时聊天链路从 API 请求线程中解耦。

建议设计：

- API 只负责：
  - 鉴权
  - 限流
  - 参数校验
  - 创建任务记录
  - 投递 Kafka 消息
- Worker 负责：
  - 消费任务
  - 执行 Agent
  - 调用工具
  - 更新任务状态
  - 写回结果

建议接口：

- `POST /chat/tasks`
- `GET /chat/tasks/{task_id}`
- `GET /chat/tasks/{task_id}/result`

第一阶段建议采用：

- “提交任务 + 轮询结果”的模式

后续可扩展：

- SSE
- WebSocket
- token 级流式事件

---

## 5.4 Kafka 接入与 worker 进程化

当前状态：进行中，已完成第一小步

已落地：

- 新增 `task_broker` 抽象，统一任务投递与消费边界
- 默认保留 `InMemory` broker，保证本地链路继续可用
- 新增 Kafka broker 可选实现，接口保持与现有任务 API 兼容
- `chat_service` 已改为投递到 broker，而不是直接依赖进程内队列
- `task_worker` 已收敛为任务消费执行器，职责不再和队列实现耦合
- API 进程仅在 `inmemory` 模式下启动内置 worker
- 新增独立 worker 启动脚本：`backend/scripts/run_task_worker.py`

这一阶段的目标是把“消息投递”和“消息执行”彻底拆开，为后续真正接入 Kafka、独立部署 worker、补重试和 DLQ 打下不会返工的基础。

## 6. Kafka 接入方案

## 6.1 Kafka 的定位

Kafka 在本项目中不用于替代普通 Web 接口，而用于承载高耗时异步任务和事件。

Kafka 主要服务于：

- 聊天任务削峰
- Agent 执行解耦
- 异步结果处理
- 审计事件扩展

不建议使用 Kafka 承载：

- 注册登录
- 用户信息查询
- 会话列表查询
- 普通同步 CRUD

---

## 6.2 建议 Topic 设计

第一阶段建议最小集合：

- `chat.request`
  - API 写入，Worker 消费
- `chat.request.dlq`
  - 多次失败后的死信队列
- `audit.event`
  - 登录、登出、任务创建、任务失败等审计事件

第二阶段可增加：

- `chat.result`
- `tool.event`
- `chat.stream.event`

---

## 6.3 Kafka 消息设计

建议每条聊天任务消息至少包含：

- `task_id`
- `user_id`
- `session_id`
- `message_id`
- `input_text`
- `request_id`
- `created_at`
- `trace_meta`

建议规则：

- 使用 `session_id` 作为消息 key，尽量保证同一会话顺序执行
- Worker 处理必须具备幂等性，避免重复消费写出重复结果
- 失败任务必须支持重试和 DLQ

---

## 6.4 Worker 执行规则

Worker 侧应具备：

- 任务超时控制
- 最大重试次数
- 死信处理
- 错误分类
- 任务状态机

建议状态机：

- `queued`
- `running`
- `succeeded`
- `failed`
- `timeout`
- `cancelled`

## 8. 并发与性能规划

## 8.1 当前瓶颈判断

本项目的主要性能瓶颈通常不在普通接口，而在：

- 模型调用
- Web 搜索工具调用
- 会话记忆读写
- 流式输出期间连接占用

因此并发治理重点不应只是“加机器”，而是：

- 解耦长任务
- 控制单用户资源占用
- 为外部依赖设置保护机制

---

## 8.2 并发治理策略

建议先采用以下策略：

- API 层限流
- 用户级限流
- 会话级串行执行
- Worker 并发数控制
- 工具调用超时与失败降级
- 任务队列削峰

推荐原则：

- 同一 `session_id` 的任务尽量串行，避免上下文混乱
- 同一用户同时运行任务数量应有限制
- 高耗时任务要有超时上限

---

## 8.3 性能目标建议

作为第一阶段可上线版本，建议目标如下：

- 普通查询接口 P95 < 300ms
- 聊天任务创建接口 P95 < 500ms
- 聊天任务进入队列成功率 >= 99.9%
- 常规聊天任务完成成功率 >= 95%
- 队列积压和超时任务可观测

## 14. 风险与注意事项

- 不建议一开始就把所有接口 Kafka 化
- 不建议在没有用户模型前先做复杂并发治理
- 不建议把 Agent 内部 memory 直接当业务聊天存储使用
- 不建议在没有日志和状态机的前提下直接上 Worker
- 不建议把“能跑通”误认为“能上线”

本项目最合理的路线是：

- 先补用户和数据模型
- 再补任务异步化
- 再补限流、安全、监控
- 最后再做更复杂的实时推送和平台化扩展

## 15. 建议的最终上线形态

如果按本计划推进，第一阶段上线版本应具备以下能力：

- 用户可以注册、登录和管理自己的聊天记录
- 用户发起聊天请求时系统会创建异步任务
- Worker 可以独立进程运行，并通过 Kafka 消费任务执行 Agent
- 结果可以被前端查询和展示
- 系统具备基础限流、安全、日志和部署能力

这会让当前项目从“单体 Agent Demo”升级为“具备真实上线基础的小型 AI Web 产品”。
